# Команды агентов: параллельные Claude строят настоящий софт

Большинство демо с агентами выглядят впечатляюще и не делают ничего полезного. Приложение со списком дел. Скрапер задач. «Персональный исследовательский ассистент», который пересказывает три веб-страницы.

А потом кто-то вставляет шестнадцать Claude в цикл, и они строят C-компилятор, способный собрать ядро Linux.

Эта статья как раз про вторую историю — и конкретно про то, как выглядел харнесс (harness, обвязка/runtime-обёртка) под ней. Мы разберём архитектуру проекта Николаса Карлини *«Building a C compiler with a team of parallel Claudes»*
([Anthropic Engineering](https://www.anthropic.com/engineering/building-c-compiler),
[GitHub](https://github.com/anthropics/claudes-c-compiler)), вытащим из неё принципы проектирования, которые действительно сыграли роль, и переведём их в паттерны, которые вы сможете переиспользовать в собственном харнессе.

---

## Главная мысль

> **16 экземпляров Claude, работавших параллельно около двух недель, выдали
> C-компилятор на 100 000 строк, написанный на Rust. Он компилирует Linux 6.9
> на x86, ARM и RISC-V. Он проходит 99% набора стресс-тестов GCC (GCC torture
> test suite). Он компилирует QEMU, FFmpeg, SQLite, Postgres и Redis.**

Перечитайте это. А потом ещё раз.

Это не игрушка. Это настоящий компилятор продакшен-уровня, написанный командой LLM при ограничениях намного более жёстких, чем стерпела бы человеческая команда: без доступа в интернет, без готовых крейтов с компиляторами, без копирования из LLVM или tcc — только стандартная библиотека Rust и то, что Claude мог вывести сам из спецификации C и собственного обучения. Чистая реализация «с нуля» (clean-room), сделанная агентами.

Интересный вопрос вовсе не «может ли LLM написать компилятор». Мы и так знали, что *части* компилятора он написать может. Интересный вопрос такой:

**Какой харнесс превращает 16 безпамятных, забывчивых вызовов LLM в слаженную инженерную команду, которая выкатывает 100 000 строк кода?**

## Архитектура Ralph-loop

Сердце харнесса до неприличия простое. Каждый агент — это bash-цикл:

```bash
#!/usr/bin/env bash
# agent.sh — run inside a Docker container
while true; do
  claude --dangerously-skip-permissions -p AGENT_PROMPT.md
done
```

Вот и всё. Это весь агент целиком.

Никакого оркестратора. Никакого демона очереди задач. Никакого планировщика. Никакой шины сообщений. Когда одна сессия Claude завершается — успехом, провалом или схлопыванием контекстного окна — цикл запускает свежий Claude с тем же файлом промпта. Промпт велит новому Claude пойти посмотреть в репозиторий, выяснить, что делать дальше, и сделать это.

Этот стиль иногда называют «Ralph loop» (в честь фразы Ральфа Виггама «I'm helping!»): тупой внешний цикл продолжает пинать умный внутренний процесс, пока работа не будет сделана.

Единственное вмешательство уровня супервизора — это:

```bash
# kill the fleet
pkill -9 bash
```

Вот и весь протокол выключения. Никакого аккуратного слива (graceful drain), никакого «пожалуйста, доделай текущую задачу». Вы шлёте SIGKILL bash-циклам, и агенты умирают на полумысли. В следующий раз, когда вы их запустите, они подхватят работу с того места, на котором их оставил `git pull`.

### Почему это работает

Ralph loop — правильная форма для агентной работы по трём причинам:

1. **LLM всё равно безпамятны между сессиями.** Делать вид, что это долгоживущие процессы, — значит лишь прятать схлопывание контекстного окна за всё более отчаянными ухищрениями.
2. **Перезапуск бесплатен.** Свежий Claude, читающий репозиторий, обходится дешевле, чем выдохшийся Claude, забивший свой контекст тремя часами провалившихся экспериментов.
3. **Изоляция отказов достаётся даром.** Если сессия сошла с рельсов, следующая не наследует обломки — она наследует состояние git, а это единственный источник истины.

## Координация через git (без оркестратора)

Вот часть, которая удивляет большинство людей: **здесь нет центрального оркестратора.** Никто не раздаёт задачи. Никто не знает, кто чем занят. Нет дашборда с надписью «agent-7 сейчас работает над struct-packing».

Вместо этого флот координируется через два примитива:

1. **Голый git-репозиторий** (bare git repo), в который каждый контейнер пушит и из которого пуллит.
2. Каталог `current_tasks/` с **lock-файлами** — по одному на каждую задачу в работе, записываемому как `current_tasks/<task-name>.txt`.

Промпт каждого агента, по сути, говорит ему:

> «Посмотри на открытые TODO, падающие тесты и файлы прогресса. Выбери самое очевидное следующее дело, над которым никто сейчас не работает. Положи на него lock-файл. Сделай работу. Закоммить. Сними lock. Выйди.»

Вот жизненный цикл одной сессии:

```
 ┌──────────────────────────────────────────────────────────────┐
 │  1. git clone / pull                                         │
 │  2. scan repo → pick task T                                  │
 │  3. git add current_tasks/T.txt   ← claim the lock           │
 │  4. git push                      ← publishes the claim      │
 │  5. work on T (write code, run tests, commit)                │
 │  6. git pull --rebase             ← grab peers' changes      │
 │  7. resolve merges / re-run tests                            │
 │  8. git push                      ← ship it                  │
 │  9. git rm current_tasks/T.txt    ← release lock             │
 │ 10. git push                                                 │
 │ 11. exit → Ralph loop spawns a new Claude in a new container │
 └──────────────────────────────────────────────────────────────┘
```

### Гонка, и почему git выигрывает

Два агента просыпаются одновременно. Оба смотрят на открытые задачи. Оба решают, что самое очевидное — «исправить segfault в codegen/x86.rs». Оба пишут `current_tasks/fix-segfault-x86.txt`. Оба делают `git push`.

Один push побеждает. Другой отклоняется — `non-fast-forward`. Этот агент делает `git pull --rebase`, видит, что lock уже существует, понимает, что опоздал к раздаче, и идёт выбирать другую задачу.

В этом весь приём: **модель конкурентности git — это же и модель конкурентности агентной команды.** Локи, мержи и разрешение конфликтов достаются бесплатно, потому что git их уже решает. Вы не строите научную статью по распределённым системам поверх своих LLM; вы наследуете готовую из 2005 года.

## Принцип проектирования 1: пиши тесты для Claude, а не для себя

Этот урок я бы вытатуировал на лбу любому, кто строит агентные харнессы.

Когда человек читает падающий тест, он многое получает даром: он может пробежать глазами 500 строк вывода, заметить подозрительный паттерн, держать гипотезу в голове и пойти отлаживать. LLM в цикле ничего из этого толком не умеет. Ваш тестовый харнесс должен быть спроектирован под его ограничения, а не под ваши.

### Засорение контекстного окна

Поведение большинства тест-раннеров по умолчанию — «вывалить всё в stdout». Это нормально для человека, который скроллит в терминале; для Claude это катастрофа. Тысяча строк тестового вывода прогрызёт контекстное окно, и к тому моменту, когда Claude начнёт рассуждать о баге, он уже забудет промпт.

**Исправление**: сделайте тестовый харнесс предельно лаконичным по умолчанию, а подробности отправляйте в файлы.

```python
def run_test(name: str) -> TestResult:
    result = execute(name)
    # Human-visible (and LLM-visible) output: one line.
    if result.ok:
        print(f"PASS {name}")
    else:
        print(f"ERROR {name}  see logs/{name}.log")
    # Everything else goes to disk.
    write(f"logs/{name}.log", result.full_output)
    return result
```

Теперь Claude видит:

```
PASS  tests/struct_layout_01
PASS  tests/struct_layout_02
ERROR tests/struct_layout_03  see logs/struct_layout_03.log
PASS  tests/struct_layout_04
```

Claude может сделать `grep ERROR` по всему прогону, выбрать одно падение, а затем `cat` только нужного лога. Контекст остаётся маленьким. Внимание остаётся сфокусированным.

### Маркер `ERROR`

Обратите внимание на буквальный `ERROR`. Это сделано намеренно. Единообразный, грепаемый маркер — одно из соглашений с самым высоким ROI в любом агентном харнессе. Он означает, что Claude может ориентироваться в тысячах результатов тестов одним `grep -l ERROR logs/`. Не пишите «failed» в одном месте, «FAIL» в другом и «❌» в третьем. Выберите одно. Поставьте его везде.

### Слепота ко времени

У Claude нет настенных часов. Он не может отличить «этот тест медленный» от «этот тест завис». Если вы позволите агенту прогонять полный набор тестов каждую сессию, половина ваших агентов будет сидеть по 40 минут, решит, что ничего не происходит, прибьёт прогон и попробует какой-нибудь дикий рефакторинг.

**Исправление**: добавьте режим `--fast` с подвыборкой, который прогоняет 1–10% тестов, выбираемых детерминированно для каждого агента:

```python
def select_tests(all_tests: list[str], agent_id: str, fraction=0.05):
    # Same agent → same subsample within a session.
    # Different agents → different subsamples, so the fleet covers
    # more ground than any single agent does.
    rng = seeded_random(agent_id)
    k = max(1, int(len(all_tests) * fraction))
    return rng.sample(all_tests, k)
```

Здесь важны два свойства:

- **Детерминированность для каждого агента**: один и тот же агент, выполняющий одну и ту же команду дважды, видит одинаковые результаты. Иначе Claude не сможет рассуждать о том, починило ли что-нибудь его изменение.
- **Различие по флоту**: каждый агент прощупывает свой угол набора тестов, и объединение их покрытия гораздо выше, чем у любого отдельного агента.

### Файлы прогресса и README

Единственный долгоживущий кусок памяти — это файловая система. Агенты Карлини ведут:

- `README.md` в начале каждого модуля: *что это делает, что работает, что нет*.
- `PROGRESS.md` и файлы прогресса по областям: *что пробовали, что провалилось, какова текущая гипотеза*.

Это не для людей. Это для следующего Claude, который проснётся. Когда вы относитесь к файлам прогресса как к **сообщениям будущему себе**, они становятся самым ценным артефактом в репозитории.

## Принцип проектирования 2: декомпозиция задач, дружелюбная к параллелизму

Параллелизм — это не факт о вашей команде. Это факт о вашей *работе*.

### Лёгкий режим: много независимых падений

Счастливый случай для агентной команды — это набор тестов с множеством независимо падающих тестов. 200 падающих тестов, 16 агентов, каждый хватает один, чинит его, коммитит, двигается дальше. Минимум конфликтов. Линейное масштабирование. Все расходятся довольными.

```
┌───────────────────────┐
│  200 failing tests    │
├───────────────────────┤
│  agent-1 → test 47    │
│  agent-2 → test 113   │
│  agent-3 → test 2     │
│  agent-4 → test 88    │
│   ...                 │
└───────────────────────┘
```

### Тяжёлый режим: одна монолитная сборка

А теперь представьте, что цель — *скомпилировать ядро Linux*. Тут одна задача. Одна сборка. Один падающий артефакт. 16 агентов одновременно начинают читать исходники ядра и предлагать правки. Они перезатирают друг друга. Они спорят, чинить баг в парсере или в codegen. Репозиторий превращается в гонку на выживание с битьём машин.

Больше агентов теперь *хуже*. Бросать всё больше Claude на одну неделимую проблему — это и есть способ сжечь $20 тысяч за вечер без всякого результата.

Исправление — не «добавить оркестратор». Исправление — **разложить монолит на независимые падения**. Что подводит нас к самой остроумной идее во всём проекте.

## GCC как оракул: бисекция ядра

Когда команда попыталась скомпилировать ядро Linux своим новым компилятором, всё ломалось эффектно и запутанно. Сотни файлов вносили вклад в один сломанный образ. Никто — ни Claude, ни человек — не мог посмотреть на «ядро не загрузилось» и сказать вам, какой из 20 000 C-файлов вскрыл баг.

Приём: **использовать GCC как заведомо исправный оракул и позволить Claude компилировать лишь подмножество**.

```
┌────────────────────────────────────────────────────────────┐
│  For each kernel .c file:                                  │
│    flip a coin biased toward GCC (say 95/5)                │
│    → if GCC: compile with GCC (trusted)                    │
│    → if Claude: compile with Claude's compiler             │
│                                                            │
│  Link everything into a kernel image. Boot it.             │
│                                                            │
│  If it boots: great. Try a larger Claude subset.           │
│  If it breaks: Claude's subset contains the bug.           │
│                 → bisect the subset, narrow down,          │
│                   identify the specific failing file,      │
│                   file a focused bug report,               │
│                   add it as an isolated failing test.      │
└────────────────────────────────────────────────────────────┘
```

Псевдокод:

```python
def locate_bad_file(c_files: list[str]) -> str:
    # Standard bisection against a known-good oracle.
    lo, hi = 0, len(c_files)
    while hi - lo > 1:
        mid = (lo + hi) // 2
        subset = c_files[lo:mid]
        image = build_kernel(subset, compiler="claude",
                             rest_compiler="gcc")
        if boots(image):
            lo = mid       # bug is in the upper half
        else:
            hi = mid       # bug is in the lower half
    return c_files[lo]
```

Как только вы локализовали баг до одного файла, вы сделали критический шаг: вы превратили монолитное падение в **независимый, воспроизводимый тест-кейс**. Теперь вы снова в лёгком режиме — конкретный падающий тест, которым может владеть один Claude.

Общая форма этого паттерна:

> **Когда задача слишком велика, чтобы её распараллелить, найдите доверенный
> оракул, прогоните основную массу работы через него, а агентам дайте
> откалывать куски от сжимающегося подмножества, пока каждое оставшееся
> падение не станет локальным.**

Это паттерн проектирования харнесса, а не паттерн компиляторов. Он применим ко всему, где есть монолитная система с доверенным эталоном, — конвейеры данных, подкомпоненты обучения моделей, генерация конфигов, даже рендеринг UI.

## Специализация ролей агентов

Все 16 агентов — это один и тот же бинарник Claude, но их промпты дают им разные *работы*. Звучит как мелочь; это не мелочь. Дать каждому агенту отдельную роль резко снижает патологию «все толпятся вокруг одного и того же падения».

Пять ролей из проекта, в вольном пересказе:

```
┌──────────────────────────┬──────────────────────────────────┐
│ Role                     │ Prompt flavor                    │
├──────────────────────────┼──────────────────────────────────┤
│ duplicate-code-cleaner   │ Find repeated patterns, extract  │
│                          │ helpers, reduce LOC without      │
│                          │ changing behavior.               │
├──────────────────────────┼──────────────────────────────────┤
│ performance-tuner        │ Profile, find hot paths, apply   │
│                          │ local optimizations. Do not      │
│                          │ change semantics.                │
├──────────────────────────┼──────────────────────────────────┤
│ codegen-optimizer        │ Focus on the backend. Improve    │
│                          │ emitted assembly quality.        │
├──────────────────────────┼──────────────────────────────────┤
│ design-critic            │ Do not write code. Read the repo │
│                          │ and file TODOs for design smell, │
│                          │ dead code, inconsistency.        │
├──────────────────────────┼──────────────────────────────────┤
│ doc-maintainer           │ Keep README / PROGRESS files     │
│                          │ honest and current.              │
└──────────────────────────┴──────────────────────────────────┘
```

Разделение ролей важно, потому что оно **меняет то, какие задачи агент считает «самыми очевидными».** `design-critic`, глядя на репозиторий, видит другую «следующую работу», чем `performance-tuner`, глядящий на тот же репозиторий. Флот покрывает большую часть поверхности работы без всякого центрального планировщика, решающего, кто что делает.

Для вашего собственного харнесса общее правило таково: **если два агента постоянно сталкиваются, обычно это потому, что их промпты не различают, как выглядит «очевидный следующий шаг».**

## Когда НЕ стоит строить команду агентов

Честная часть статьи. Команды агентов дороги, неуклюжи и временами волшебны. Для большинства работы это неправильный инструмент. *Не* стройте такую команду, если выполняется хоть что-то из этого:

- **Проект маленький.** Если одна сфокусированная сессия Claude может закончить задачу за один проход, флот из 16 наплодит накладные расходы на координацию, конфликты мержа и счёт на $500 за работу на $5. Один острый нож лучше шестнадцати тупых.
- **Работа требует тесной координации.** Дизайны с множеством сквозных решений — схемы API, инварианты протокола, общие модели данных — разносятся параллельными агентами в щепки. Агенты хороши в «иди почини этот падающий тест». Они плохи в «давайте договоримся, какой должна быть иерархия типов ошибок».
- **У вас нет хорошего тестового оракула.** У компилятора есть набор стресс-тестов GCC и ядро, которое либо грузится, либо нет. У большинства софта нет доверенного, автоматизируемого, детерминированного сигнала «сработало ли?». Без него ваши агенты оптимизируют под то, что проверяет верификатор, а не под то, что вам на самом деле нужно, — и поскольку верификатор несовершенен, они оптимизируют не то.
- **Вы не можете дёшево перезапускаться.** Ralph-loop предполагают, что перезапуск бесплатен. Если вашим агентам надо 30 минут разогреваться, прежде чем заняться полезной работой, цикл плохо амортизируется, и флот проигрывает одному долгоживущему процессу.

Если что-то из этого применимо — возьмите один Claude с хорошим промптом и остановитесь.

## Чему это учит создателей харнессов

Отступив от конкретного проекта, чисто переносятся пять идей:

1. **Git — это координационный примитив агентов.** Не стройте очередь задач, сервис локов, шину сообщений. Используйте git. Локи, мержи, разрешение конфликтов, история, откат — вы получаете всё это бесплатно, и придумали это люди, которые уже крепко подумали о конкурентности.
2. **Качество набора тестов — потолок способностей агента.** Claude может быть лишь настолько хорош, насколько хорош сигнал, который вы ему даёте. Вкладывайтесь непропорционально много в тесты, оракулы и верификаторы. Посредственный агент с отличным набором тестов бьёт отличного агента с посредственным набором тестов каждый раз.
3. **Проектируйте харнесс под Claude, а не под себя.** Лаконичный вывод по умолчанию. Маркеры `ERROR`. Полные логи в файлы. Детерминированная подвыборка. Файлы прогресса как письма будущим сессиям. Каждое из этого существует потому, что агент отличается от вас — более забывчив, легче отвлекается, слеп ко времени.
4. **Раскладывайте монолиты с помощью заведомо исправных оракулов.** Когда проблема сопротивляется параллелизму, не добавляйте агентов; добавьте доверенный эталон, прогоните через него основной объём работы, а агентам дайте атаковать сжимающийся остаток. GCC-как-оракул — общий паттерн.
5. **Специализируйте через промпты.** Одна модель, разные работы. Самый дешёвый и лучше всего масштабируемый способ заставить 16 агентов не наступать друг другу на ноги — дать им разные определения «очевидного следующего шага».

Счёт на $20 000 оплатил компилятор. Бесплатным бонусом оказался чертёж того, как работают команды агентов, когда они работают.
