# Оркестрация мультиагентных систем

> **Ключевая идея:** Одиночный агент упирается в жёсткие пределы — размер контекстного окна, отсутствие доменной специализации и последовательное исполнение. Мультиагентная оркестрация (multi-agent orchestration) распределяет работу между несколькими агентами, у каждого из которых есть собственный контекст и собственные инструменты, а координируются они паттернами, заимствованными из распределённых систем.

Статья про [Суб-агент](sub-agent.md) разбирает простейший случай: один ведущий агент порождает воркеров под параллельные задачи. Эта статья идёт дальше — к паттернам оркестрации для систем, где несколько агентов сотрудничают, специализируются и масштабируются.

## Зачем нужна мультиагентность?

Один агент, прогоняющий один Агентный цикл (Agentic Loop), — это вариант по умолчанию. Для большинства задач он работает хорошо. Но рано или поздно за его пределы вас выталкивают три стены:

### 1. Пределы контекстного окна

Даже с окном на 200K токенов сложный проект способен исчерпать доступный контекст. Рефакторинг кодовой базы, затрагивающий 80 файлов; исследовательская задача, поглощающая 40 исходных документов; многошаговый конвейер, накапливающий выводы инструментов, — всё это может переполнить одно окно. Когда контекст заполняется, модель начинает забывать ранние инструкции и теряет связность.

Мультиагентность разбивает работу на части. Каждый агент получает свежее контекстное окно, выделенное под его срез задачи.

### 2. Специализация

Универсальный агент, который пишет код, ревьюит PR, генерирует маркетинговые тексты и проводит анализ данных, посредственен во всём перечисленном. Агенты работают лучше, когда их системный промпт, инструменты и контекст настроены под конкретный домен. «Агент код-ревью» с загруженными линтерами и руководствами по стилю даёт более качественные ревью, чем агент общего назначения, у которого вдобавок просто есть доступ к линтеру.

### 3. Параллелизм

Одиночный агент работает последовательно. Он вызывает один инструмент, ждёт результат, рассуждает, вызывает следующий инструмент. Когда у вас есть пять независимых задач — написать тесты для пяти модулей, исследовать пять конкурентов, перевести на пять языков, — одиночный агент обрабатывает их по очереди. Пять агентов обрабатывают их одновременно.

## Паттерны оркестрации

Четыре паттерна покрывают большинство мультиагентных архитектур. Они не взаимоисключающие — в продакшен-системах их часто комбинируют.

### Последовательный конвейер (Sequential Pipeline)

Агент A завершает свою работу и передаёт результат агенту B, который передаёт свой результат агенту C. Каждый агент преобразует или обогащает результат.

```
┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐
│ Ingest  │────►│ Analyze │────►│  Draft  │────►│ Review  │
│  Agent  │     │  Agent  │     │  Agent  │     │  Agent  │
└─────────┘     └─────────┘     └─────────┘     └─────────┘
```

**Когда использовать:** Задачи с естественными стадиями, где каждая стадия требует иной экспертизы или иных инструментов. Пример: собрать данные → очистить и проанализировать → сгенерировать отчёт → вычитать и проверить факты.

**Ключевое свойство:** Каждый агент видит только выход предыдущей стадии, а не весь накопленный контекст. Это преимущество — оно вынуждает делать чёткие интерфейсы между стадиями и предотвращает разрастание контекста.

### Fan-Out / Fan-In

Координатор раздаёт одну и ту же (или связанные) задачу N агентам параллельно, а затем собирает и сводит все результаты.

```
                    ┌───────────┐
               ┌───►│ Worker A  │───┐
               │    └───────────┘   │
┌──────────┐   │    ┌───────────┐   │    ┌──────────┐
│Dispatcher│───┼───►│ Worker B  │───┼───►│  Merger  │
└──────────┘   │    └───────────┘   │    └──────────┘
               │    ┌───────────┐   │
               └───►│ Worker C  │───┘
                    └───────────┘
```

**Когда использовать:** Задачи, которые декомпозируются на независимые, распараллеливаемые подзадачи. Пример: перевести документ на 8 языков, написать модульные тесты для 10 модулей, исследовать 5 конкурирующих продуктов.

**Ключевое свойство:** Воркеры никак не взаимодействуют друг с другом. Сводящий агент (merger) — или сам диспетчер — отвечает за объединение результатов и разрешение конфликтов.

### Супервизор (Supervisor)

Один надзорный агент принимает все решения о делегировании. Он наблюдает текущее состояние, решает, какого специалиста вызвать следующим, читает результат и решает, что делать дальше. В отличие от конвейера, супервизор может зацикливаться, переделегировать или вызывать одного и того же специалиста несколько раз — в зависимости от промежуточных результатов.

```
                 ┌─────────────────┐
                 │   Supervisor    │
                 │ (decides, loops)│
                 └───┬───┬───┬────┘
                     │   │   │
            ┌────────┘   │   └────────┐
            ▼            ▼            ▼
      ┌──────────┐ ┌──────────┐ ┌──────────┐
      │ Code     │ │ Research │ │ Review   │
      │ Agent    │ │ Agent    │ │ Agent    │
      └──────────┘ └──────────┘ └──────────┘
```

**Когда использовать:** Задачи, где следующий шаг зависит от результата предыдущего, а решение о том, *какого* специалиста вызвать, требует суждения. Пример: координирующий агент по программированию, который делегирует реализацию агенту-кодеру, отправляет результат агенту-ревьюеру и возвращается к агенту-кодеру, если ревью нашло проблемы.

**Ключевое свойство:** Супервизор держит общий план и состояние. Специалисты не имеют состояния (stateless) — они получают задачу, возвращают результат и забывают её. Это сохраняет контекстные окна специалистов чистыми.

### Одноранговая схема (Peer-to-Peer)

Агенты общаются напрямую друг с другом без центрального координатора. У каждого агента свой входящий ящик, и он может отправлять сообщения любому другому агенту.

**Когда использовать:** Симуляции, рассуждения в стиле дебатов (агент A аргументирует «за», агент B — «против», агент C судит) или системы, где агенты представляют разных заинтересованных сторон.

**Ключевое свойство:** Нет единой точки координации. Это делает одноранговую схему самым гибким паттерном, но и самым трудным для отладки. Без супервизора сложно гарантировать завершение работы или отслеживать прогресс.

**Предупреждение:** В продакшен-системах на базе харнесса (Harness) одноранговая схема редко оказывается верным выбором. Отсутствие центрального управления значительно усложняет обработку ошибок, контроль таймаутов и сбор результатов. Предпочитайте Супервизор или Fan-Out/Fan-In, если только у вас нет конкретной причины для децентрализованной коммуникации.

## Реализация в харнессе

Харнесс (harness) — программа-хост, оборачивающая LLM в работающего агента, см. [Что такое харнесс](what-is-harness.md) — реализует мультиагентную оркестрацию через четыре механизма: порождение суб-агентов, изоляцию контекста, коммуникацию по принципу «родитель читает потомка» и обработку таймаутов.

### Порождение суб-агентов

Базовый примитив — порождение изолированной сессии агента. В OpenClaw это паттерн `sessions_spawn`:

```
Parent Agent
    │
    ├── sessions_spawn(label="research", task="Find competitor pricing")
    │       → creates isolated session with own context, tools, model
    │
    ├── sessions_spawn(label="writer", task="Draft the pricing page")
    │       → another isolated session
    │
    └── waits for both to complete (push-based)
```

Каждая порождённая сессия — полностью независимый агент со своими собственными:
- Контекстным окном (нет общего бюджета токенов с родителем)
- Системным промптом (можно настроить под каждый суб-агент)
- Набором инструментов (можно ограничить или расширить)
- Моделью (для более простых задач можно взять модель подешевле)

Родитель задаёт метку (label), описание задачи и опционально модель или уровень «размышления» (thinking level). Харнесс берёт на себя жизненный цикл процесса, таймаут и доставку результата.

### Изоляция контекста

Изоляция контекста — это не удобство, а основное проектное ограничение. Контекстное окно каждого суб-агента полностью независимо:

```
┌──────────────────────┐    ┌──────────────────────┐
│   Parent Context     │    │  Sub-Agent Context    │
│                      │    │                       │
│ [system prompt]      │    │ [system prompt]       │
│ [user conversation]  │    │ [task instruction]    │
│ [tool results A,B,C] │    │ [tool results X,Y]   │
│ [128K tokens used]   │    │ [12K tokens used]     │
│                      │    │                       │
│  Cannot see ────────►│ X  │◄──── Cannot see       │
│  sub-agent context   │    │  parent context       │
└──────────────────────┘    └──────────────────────┘
```

**Почему это важно:** Родительский агент со 128K токенов накопленного диалога может породить суб-агент, который стартует с чистым бюджетом в 200K токенов. Суб-агент не отягощён нерелевантной историей — он видит только инструкцию своей задачи и тот контекст, который родитель явно ему передал.

Следствие: суб-агенты не могут читать переменные родителя, выводы его инструментов или историю диалога. Вся передача информации происходит явно — родитель включает релевантный контекст прямо в строку инструкции к задаче.

### Коммуникация: родитель читает результаты потомка

Мультиагентная коммуникация в харнессе следует строгому паттерну: **родитель читает результаты потомка; потомки не пишут в память родителя.**

```
1. Parent spawns child with task instruction (one-way write)
2. Child executes independently
3. Child completes → Harness delivers result to parent (push-based)
4. Parent reads the result in its next Agentic Loop iteration
```

Нет ни общей памяти, ни шины сообщений, ни механизма обратного вызова, через который потомок изменял бы состояние родителя. Это ограничение радикально упрощает рассуждения о системе — родитель всегда остаётся единственным источником истины об общем прогрессе.

В OpenClaw завершение работы устроено **по принципу push** (push-based): когда суб-агент заканчивает, результат автоматически объявляется в сессии родителя. Родителю не нужно опрашивать. Это устраняет самую частую ошибку мультиагентных систем: плотные циклы опроса (polling), которые впустую тратят токены, проверяя «ну что, уже готово?».

### Обработка таймаутов и ошибок

Суб-агенты могут зависать, падать или выдавать мусор. Продакшен-харнесс обязан справляться со всеми тремя случаями:

```python
@dataclass
class SubAgentConfig:
    task: str
    label: str
    timeout_seconds: int = 300       # Kill after 5 minutes
    max_retries: int = 1             # Retry once on failure
    fallback: str | None = None      # Alternative task on repeated failure

class SubAgentManager:
    def spawn_with_safeguards(self, config: SubAgentConfig) -> str:
        for attempt in range(config.max_retries + 1):
            try:
                result = self.spawn(
                    task=config.task,
                    label=config.label,
                    timeout=config.timeout_seconds,
                )
                if self.validate_result(result):
                    return result
                # Invalid result — retry with clarified instructions
                config.task += "\n\nPrevious attempt produced invalid output. Be precise."
            except TimeoutError:
                if attempt == config.max_retries and config.fallback:
                    return self.spawn(task=config.fallback, label=f"{config.label}-fallback")
                continue
            except Exception as e:
                log.error(f"Sub-agent {config.label} failed: {e}")
                if attempt == config.max_retries:
                    return f"[FAILED] {config.label}: {e}"
        return f"[FAILED] {config.label}: max retries exceeded"
```

**Ключевые принципы:**
- **Всегда задавайте таймауты.** Суб-агент без таймаута — это утечка ресурсов, которая только и ждёт своего часа.
- **Валидируйте результаты.** Суб-агент, который «завершился» с пустым выводом или сообщением об ошибке, — это не успех.
- **Деградируйте плавно.** Когда суб-агент падает, родителю стоит продолжить с частичными результатами, а не прерывать всё целиком — если только провалившаяся задача не была критичной.
- **Ограничивайте глубину.** Суб-агенты, порождающие собственных суб-агентов, дают экспоненциальный рост сложности. Ограничивайте рекурсию 1–2 уровнями.

## Примеры из реальной жизни

Три open-source-проекта демонстрируют мультиагентную оркестрацию на разных масштабах.

### Multica (14K+ звёзд)

**Паттерн:** Супервизор + Fan-Out / Fan-In

Multica относится к агентам как к участникам команды на доске проекта. Человек (или ведущий агент) назначает GitHub-issues агентам-воркерам, каждый из которых работает независимо. Система отслеживает прогресс на доске в стиле Kanban, показывая, какой агент над чем работает, что находится на ревью, а что уже смержено.

Ключевые проектные решения:
- **Поддержка нескольких рантаймов** — агенты могут работать на Claude Code, Codex, OpenClaw или OpenCode. Оркестратору безразлично, какой рантайм использует воркер, важно лишь, чтобы тот выдал ожидаемый результат.
- **Составные навыки** — агенты могут комбинировать несколько навыков (skills); например, агент «fix-and-test» загружает и навык программирования, и навык тестирования.
- **Делегирование на основе issues** — работа организована вокруг GitHub-issues, а не абстрактных «задач». Это даёт каждому агенту конкретную, отслеживаемую единицу работы с ясными критериями приёмки.

### Paseo (3.6K+ звёзд)

**Паттерн:** Супервизор с межустройственным исполнением

Paseo предоставляет единый унифицированный интерфейс для взаимодействия с несколькими агентами. Вы отдаёте команду, и Paseo маршрутизирует её нужному агенту — или раздаёт её нескольким агентам для параллельного исполнения.

Ключевые проектные решения:
- **Один интерфейс, много агентов** — пользователю не нужно знать, какой агент обрабатывает какую задачу. Решает слой маршрутизации Paseo.
- **Межустройственность** — одна и та же оркестрация работает с телефона, десктоп-приложения или CLI. Агенты выполняются на стороне сервера; интерфейс — лишь тонкий клиент.
- **Параллелизм по умолчанию** — когда задачи независимы, Paseo автоматически распараллеливает их между доступными агентами, а не ставит в очередь.

### OpenClaw

**Паттерн:** Нативное порождение суб-агентов с завершением по принципу push

OpenClaw реализует мультиагентную оркестрацию как полноценный примитив харнесса через `sessions_spawn`. Родительский агент порождает суб-агентов одним вызовом инструмента, указывая метку, задачу и опциональную конфигурацию модели/уровня размышления.

Ключевые проектные решения:
- **Завершение по принципу push** — когда суб-агент заканчивает, его результат автоматически объявляется в сессии родителя. Опрос не требуется. Это явно задокументировано: *«Completion is push-based: it will auto-announce when done.»*
- **Отслеживание по меткам** — каждый суб-агент получает человекочитаемую метку (например, `article-research`, `code-review`). Родитель может перечислять суб-агентов, направлять их или завершать по метке.
- **Ограничение глубины** — суб-агентам можно задать максимальную глубину, чтобы предотвратить неконтролируемое рекурсивное порождение.
- **Инъекция контекста** — родитель может внедрить конкретный контекст (файлы, инструкции, ограничения) в сессию суб-агента, не делясь всей своей историей диалога.

## Когда НЕ стоит использовать мультиагентность

Мультиагентная оркестрация имеет реальную цену: рост задержек (накладные расходы на порождение), повышенный расход токенов (дублирование системных промптов и контекста между агентами) и дополнительную сложность отладки. Избегайте её, когда:

### Задача умещается в одно контекстное окно

Если задача занимает 10K токенов контекста и 5 вызовов инструментов, одиночный агент справится быстрее и надёжнее, чем порождение воркеров. Накладные расходы на мультиагентную координацию (порождение, передача контекста, слияние результатов) превышают выгоду.

### Стоимость координации превышает выгоду

Задача, требующая тесного обмена «туда-сюда» между двумя агентами — где агенту A нужен результат агента B, прежде чем он сделает следующий шаг, и наоборот, — лучше решается одиночным агентом, делающим последовательные вызовы инструментов. Мультиагентность блистает, когда работа *независима*; она пробуксовывает, когда работа глубоко переплетена.

### Требуется строгий порядок

Если шаги должны выполняться в точном порядке с прецизионными передачами, а возможности для параллелизма нет, одиночный агент с пошаговым планом проще и детерминированнее. Последовательный конвейер из агентов добавляет задержку на каждой границе стадий без какого-либо выигрыша в параллелизме — используйте его только тогда, когда стадии действительно выигрывают от изоляции контекста или специализации.

## Антипаттерны

### Разделяемое изменяемое состояние

Два агента одновременно пишут в один и тот же файл, строку базы данных или переменную. Это порождает состояния гонки, затирание работы и недетерминированные результаты. Исправление: закрепите за каждым агентом владение отдельными ресурсами. Если двум агентам всё же нужно внести вклад в один артефакт, используйте паттерн Fan-Out/Fan-In, где сводящий агент (merger) объединяет их выводы последовательно.

### Неограниченный Fan-Out

Порождение агентов пропорционально размеру входных данных без ограничения. «Переведи этот документ на все 50 поддерживаемых языков» порождает 50 одновременных агентов, перегружая систему. Исправление: задайте лимит `max_workers` и обрабатывайте пакетами.

```python
# Bad: unbounded
for lang in all_50_languages:
    spawn_agent(task=f"Translate to {lang}")

# Good: bounded batches
BATCH_SIZE = 8
for batch in chunked(all_50_languages, BATCH_SIZE):
    results = fan_out_fan_in(
        [f"Translate to {lang}" for lang in batch],
        merge_prompt="Collect all translations.",
    )
```

### Отсутствие таймаута или предохранителя

Суб-агент, который завис — ожидает API, упёршийся в rate-limit, застрял в бесконечном цикле или просто выполняет сложную задачу, — блокирует всю оркестрацию, если таймаута нет. Каждое порождение суб-агента обязано включать:
- **Таймаут**, завершающий агент по истечении максимальной длительности
- **Предохранитель** (circuit breaker), прекращающий повторы после N сбоев
- **Запасной путь** (fallback), позволяющий родителю продолжить с частичными результатами

### Чрезмерная декомпозиция

Разбиение 30-секундной задачи на 5 суб-агентов, каждый из которых тратит по 15 секунд накладных расходов на порождение и возврат результатов. Суммарное «настенное» время вырастает с 30 секунд до 75. Мультиагентность — инструмент управления сложностью и параллелизмом, а не значение по умолчанию для каждой задачи.

## Как выбрать правильный паттерн

| Ситуация | Паттерн | Причина |
|-----------|---------|--------|
| Стадии с разной экспертизой | Последовательный конвейер | Каждая стадия получает специализированный контекст и инструменты |
| N независимых одинаковых задач | Fan-Out / Fan-In | Максимальный параллелизм, простое слияние |
| Сложная задача, требующая суждения о следующих шагах | Супервизор | Централизованное принятие решений, гибкая маршрутизация |
| Дебаты или многоперспективный анализ | Одноранговая схема | Агенты представляют разные точки зрения |
| Смесь параллельной и последовательной работы | Супервизор + Fan-Out | Супервизор решает, когда распараллеливать |

Начинайте с простейшего паттерна, который работает. Одиночный агент с грамотным использованием инструментов покрывает 80% задач. Когда вы упрётесь в пределы, описанные в разделе [Зачем нужна мультиагентность?](#зачем-нужна-мультиагентность), берите самый лёгкий паттерн оркестрации, который решает конкретное узкое место. Статья про [Суб-агент](sub-agent.md) разбирает простейший паттерн делегирования — начните с него, прежде чем строить полноценный слой оркестрации.
