# Initializer + Coding Agent — двухфазный паттерн харнесса

Если вы хоть раз давали топовой кодинг-модели промпт вроде *«собери мне клон Claude.ai»* и уходили на полдня, то знаете, чем это заканчивается: репозиторий с рабочей формой логина, наполовину готовым списком сообщений, `TODO: hook up streaming` где-то посреди файла и очень уверенным сообщением коммита. У модели закончился контекст.

Первый порыв — обвинить модель. Но почти всегда виноват харнесс (harness, обвязка/runtime-обёртка).

Этот пост — про конкретный паттерн, который устраняет целый класс таких сбоев: **разбейте своего долгоживущего агента (long-running agent) на две фазы — Initializer, который запускается один раз, и Coding Agent, который запускается многократно, причём каждая сессия делает ровно одну фичу с чистого листа.**

Он основан на статье [Effective harnesses for long-running agents](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents), а также на референсе из [autonomous-coding quickstart](https://github.com/anthropics/claude-quickstarts/tree/main/autonomous-coding). Я хочу разобрать, *почему* он работает, а не только как он выглядит.

## Ключевая мысль: сжатия недостаточно

Стандартный ответ на «окна контекста конечны» — сжатие (compaction). Резюмируем старое, оставляем новое, крутим цикл бесконечно.

Сжатие нормально работает для двухчасового рефакторинга. И разваливается на двухдневной сборке.

Натравите Opus 4.5 на Agent SDK на высокоуровневый промпт вроде «собери клон Claude.ai» — и понаблюдайте:

- Сессия 1 пишет кучу кода и упирается в лимит контекста с наполовину доделанными фичами.
- Сессия 2 читает сжатое резюме — это версия произошедшего с потерями, поданная как нарратив, — и вынуждена восстанавливать из него исходный замысел.
- Сессия 3 читает сжатие сжатия. Информация деградирует геометрически.

К четвёртой сессии модель играет в «испорченный телефон» сама с собой из прошлого. Она не знает, какие фичи готовы, какие наполовину готовы, а какие она просто нагаллюцинировала в резюме.

**Сжатие — это память внутри контекста (in-context memory). Долгоживущим агентам нужна память вне контекста (out-of-context memory)** — файлы, структурированное состояние, история git. То, что агент перечитывает каждую сессию, а не то, что он тащит вперёд через резюмирование.

Это и есть главный тезис. Всё остальное — механика.

## Два режима отказа, с которыми вы столкнётесь

Прежде чем перейти к паттерну, конкретизируем, что идёт не так без него. Доминируют два режима отказа.

### (A) Склонность к one-shot

Модель пытается сделать всё за одну сессию. Получив большой промпт, она пишет большое решение. Контекст заполняется. Она продолжает наседать. Сессия заканчивается с наполовину реализованными фичами, `// TODO`, рассыпанными посреди функций, набором тестов, который не запускается, и без чистого коммита, к которому можно вернуться. Модель понятия не имеет, что всё сломано, — у неё закончилось место раньше, чем она успела проверить.

### (B) Преждевременная победа

Хитрее. Сессия 2 читает заметки о прогрессе из сессии 1: «реализованы логин, сообщения, стриминг». Модель смотрит на код, видит нечто, что *похоже* на логин + сообщения + стриминг, и объявляет проект завершённым. Она его не запускала. Она не тестировала сценарий. Она поверила нарративу.

Преждевременная победа — двойственна one-shot. One-shot проваливается, потому что модель пытается сделать слишком много; преждевременная победа проваливается, потому что следующая модель доверяет оптимизму предыдущей. Любой харнесс для долгоживущей работы должен решать **обе** проблемы сразу. Именно поэтому наивное «просто крути цикл со сжатием» не годится — сжатие усиливает нарратив предыдущей сессии, что ровно неверно, когда этот нарратив чрезмерно оптимистичен.

## Двухфазный харнесс

Исправление — структурное. Один агент не делает всё. У вас две роли — с разными зонами ответственности и разным временем жизни.

```
 ┌──────────────────────────────────────────────────────────────┐
 │                        HARNESS DRIVER                        │
 │                     (your Python script)                     │
 └──────────────────────────────────────────────────────────────┘
                  │                              │
                  │ runs once                    │ runs N times
                  ▼                              ▼
 ┌──────────────────────────┐     ┌──────────────────────────────┐
 │    INITIALIZER AGENT     │     │        CODING AGENT          │
 │                          │     │                              │
 │ • read spec / brief      │     │ • pwd + read git log         │
 │ • write init.sh          │     │ • read claude-progress.txt   │
 │ • write feature_list.json│     │ • read feature_list.json     │
 │ • write progress file    │     │ • pick highest-priority unit │
 │ • git init + first commit│     │ • run init.sh (dev server)   │
 │                          │     │ • implement ONE feature      │
 │                          │     │ • e2e test via browser       │
 │                          │     │ • flip passes: true          │
 │                          │     │ • commit + update progress   │
 └──────────────────────────┘     └──────────────────────────────┘
              │                                  ▲
              │                                  │
              └───── clean state repo ───────────┘
                     (shared on disk)
```

Initializer запускается **один раз**. Его задача — превратить расплывчатый человеческий бриф в конкретный, машиночитаемый план плюс работоспособный каркас.

Coding-агент запускается **многократно**, и каждый вызов — это свежий процесс без какой-либо памяти о предыдущем. Свою память он получает, читая файлы, которые предыдущие сессии записали на диск.

Два агента, две цели, одна общая файловая система.

## Initializer-агент

Initializer — единственная фаза, которая мыслит о проекте целиком. После того как он завершится, ни у кого больше нет глобального обзора — каждая кодинг-сессия работает локально.

Ровно четыре артефакта:

1. **`init.sh`** — единственный скрипт, который поднимает проект из холодного чекаута: ставит зависимости, прогоняет миграции, запускает dev-сервер в фоне, печатает URL. Каждая кодинг-сессия запускает его первым. Если `init.sh` сломан, всё остальное не имеет значения.
2. **`feature_list.json`** — декомпозированный бэклог. Детали формата — в следующем разделе.
3. **`claude-progress.txt`** — короткие прозаические заметки о том, где сейчас проект и что дальше. Обновляются в конце каждой сессии.
4. **Начальный git-коммит** — каркас, скрипты, список фич и файл прогресса, все на `HEAD`. С этого момента каждая сессия заканчивается коммитом. `git log` — источник истины о том, что реально поехало в продакшен.

Initializer не реализует фичи. Он не подбирает цвета. Он накрывает на стол.

## Проектирование списка фич: почему JSON лучше Markdown

«Это же просто список дел, почему бы не взять Markdown-чеклист?»

Потому что когда вы даёте Claude Markdown-файл, он трактует его как прозу. Проза редактируема. Модель — из самых лучших побуждений — перепишет ваши приоритеты, переформулирует описания, объединит две фичи, потому что «так вышло чище», и разобьёт одну на пять, потому что «это проясняет замысел». Markdown-списки дел — это зыбучие пески для агентов.

JSON — другое дело. Модели обучены трактовать JSON как структурированные данные — нечто, из чего вы читаете и куда пишете конкретные поля, а не то, что можно небрежно переписать. В сочетании с явным правилом, что записываемо только одно поле, JSON становится почти пуленепробиваемым внешним состоянием.

Разумная схема:

```json
{
  "project": "claude-ai-clone",
  "created_at": "2026-04-19T10:00:00Z",
  "features": [
    {
      "id": "auth-01",
      "title": "Email + password sign up",
      "description": "User can create account with email + password. Passwords hashed with bcrypt. Email uniqueness enforced.",
      "priority": 1,
      "depends_on": [],
      "acceptance": [
        "POST /api/signup returns 201 with session cookie",
        "duplicate email returns 409",
        "password stored as bcrypt hash, not plaintext"
      ],
      "passes": false
    },
    {
      "id": "auth-02",
      "title": "Email + password sign in",
      "description": "User can log in with credentials from auth-01.",
      "priority": 2,
      "depends_on": ["auth-01"],
      "acceptance": [
        "POST /api/signin with valid creds returns 200 + cookie",
        "wrong password returns 401",
        "signed-in user can load /app"
      ],
      "passes": false
    },
    {
      "id": "chat-01",
      "title": "Minimal chat page with message list",
      "description": "Authenticated user sees a chat page with past messages rendered from the DB.",
      "priority": 3,
      "depends_on": ["auth-02"],
      "acceptance": [
        "GET /chat renders <ul> of messages",
        "unauthenticated GET /chat redirects to /signin"
      ],
      "passes": false
    }
  ]
}
```

Эта дисциплина зашита в системный промпт (system prompt) кодинг-агента:

> Вы можете изменять только поле `passes`. Недопустимо удалять или редактировать тесты, критерии приёмки, описания или структуру этого файла. Если вы считаете, что фича сформулирована неверно, — остановитесь и сообщите об этом, не переписывайте её молча.

Эта формулировка — не перестраховка, она несущая. «Недопустимо» читается моделью как жёсткое правило. На практике этого достаточно, чтобы файл оставался стабильным на протяжении десятков сессий.

`passes: false` — вторая критически важная деталь. Это единственный рычаг агента. Когда сессия реализует фичу, проходит e2e-тесты и делает чистый коммит, она переключает булево значение на `true`. Одно поле — не проза, не «вайбы» — вот как будущие сессии узнают, что реально сделано.

## Кодинг-агент: ритуал запуска, а не вольница

Каждый раз, когда кодинг-агент просыпается, он выполняет один и тот же ритуал из пяти шагов, прежде чем притронуться к коду. Не опционально.

```
1. pwd                         → confirm we're in the project
2. git log --oneline -20       → what shipped, most recent first
   cat claude-progress.txt     → narrative notes from last session
3. cat feature_list.json       → what's done, what's next, what's blocked
4. bash init.sh                → install, migrate, start dev server
5. hit the app in a browser    → verify last session's work still works
```

Только после шага 5 агент выбирает фичу и начинает писать код.

Назовём это **контекстным бутстрапом (context bootstrap)** — внеконтекстным эквивалентом сжатия. Вместо того чтобы наследовать резюме, агент восстанавливает свой рабочий контекст из файлов на диске. Одни и те же канонические источники, в одном и том же порядке, каждую сессию.

Несколько моментов, которые стоит зафиксировать:

- **Git log раньше заметок о прогрессе.** Git — это истина в последней инстанции; заметки о прогрессе оптимистичны. Если они расходятся, побеждает git.
- **`init.sh` не подлежит обсуждению.** Если dev-сервер не запускается, ничто из заявленного предыдущей сессией не имеет значения.
- **E2E-верификация раньше выбора новой фичи.** Именно это убивает преждевременную победу. Вы не доверяете «готово» от предыдущей сессии — вы открываете браузер.

На шаге 3 агент выбирает *единственную фичу с наивысшим приоритетом, у которой `passes: false` и удовлетворены все зависимости*. Не две. Не «кластер». Одну. Затем останавливается, когда эта одна зелёная и закоммичена.

Принцип «одна фича за сессию» убивает оба режима отказа сразу: он убивает one-shot, потому что контракт искусственно ограничивает объём работы, и убивает преждевременную победу, потому что «готово» не объявляется самопровозглашённо — это e2e-тест, проходящий на чётко очерченной единице работы.

## End-to-end-тестирование — это слой доверия

Юнит-тесты необходимы, но недостаточны. Кодинг-агент их сам и написал — он не может быть единственным, кто их проверяет. Если модель неверно поняла спецификацию, юнит-тесты будут ошибочны той же формы, что и реализация, и они пройдут так, что это ничего не значит.

End-to-end — вот где вы получаете независимый сигнал. Поднимите настоящее приложение, погоняйте его как пользователь, посмотрите, что выйдет.

На практике это означает браузерную автоматизацию. [Puppeteer MCP](https://github.com/modelcontextprotocol/servers) хорошо подходит — агент может навигировать, заполнять формы, кликать и делать утверждения по DOM.

Проход верификации для `auth-01`:

```
→ navigate to http://localhost:3000/signup
→ fill #email with "test+{{timestamp}}@example.com"
→ fill #password with "correct-horse-battery-staple"
→ click button[type=submit]
→ assert url === "/app"
→ assert response cookie "session" is set
→ navigate to /signup again with same email
→ fill + submit
→ assert error text contains "already exists"
```

Три момента: (1) он прогоняется против настоящего сервера, который поднял `init.sh`, а не против мока; (2) он проверяет наблюдаемое поведение — URL, DOM, куки — а не внутренние функции; (3) он достаточно дешёв, чтобы запускать его каждую сессию, поэтому проверка из бутстрапа «выжила ли работа предыдущей сессии?» — это заодно и e2e-проход по уже зелёным фичам.

Именно это придаёт `passes: true` смысл.

## Коммиты в чистом состоянии: git + файл прогресса вместе

В конце каждой кодинг-сессии последовательно происходят две вещи:

1. `git commit -m "feat(auth-01): email+password signup"` — код приземляется.
2. В `claude-progress.txt` дописывается короткая запись — что поехало, что хитро устроено, на что следующей сессии стоит посмотреть в первую очередь.

Они делают разную работу. Git — это слой **истины**: только дополнение, с таймстампами, диф не обсуждается. `claude-progress.txt` — это слой **подсказок**: заметки для следующего агента вроде «шаблоны писем лежат в `/lib/email/`, я обезьяньим патчем подменил SMTP-клиент для тестов, остерегайся гонки в `/api/signin`, я обошёл её повтором (retry)».

Оба читаются в начале каждой сессии. Ни один не самодостаточен по отдельности. Git без подсказок лаконичен, и из него трудно восстановить замысел. Подсказки без git дрейфуют. Вместе они — та самая внеконтекстная память, которая заменяет сжатие.

Оба должны оставлять сессию в *чистом* состоянии — никаких незакоммиченных изменений, никаких наполовину доделанных файлов, никакого сломанного dev-сервера. Если фича не доделана к моменту, когда упёрлись в лимит контекста, **сессия откатывается к последнему чистому коммиту и сообщает «не готово»** вместо того, чтобы коммитить сломанную работу. Именно так вы сохраняете git заслуживающим доверия.

## Где этот паттерн уместен (а где нет)

Будьте честны относительно формы вашего проекта, прежде чем хвататься за этот паттерн.

**Хорошо подходит:** веб-приложения (декомпозируемы на фичи, поддаются e2e-тестированию), CLI (каждая подкоманда — естественная единица), внутренние инструменты / дашборды / админ-панели, конвейеры данных с дискретными стадиями.

**Плохо подходит:** одноалгоритмные исследования вроде «спроектируй механизм внимания получше» (не декомпозируется на `passes: true/false`), сильно связанные кодовые базы, где трогание одной функции вынуждает менять всё подряд, и вообще всё, чья ценность живёт в *отношениях* между компонентами — корректность компилятора заключена в глобальных инвариантах, а не в «добавить лексер» + «добавить парсер».

Эмпирическое правило: если вы можете сформулировать приёмку как «пользователь (или шелл) делает X и наблюдает Y», этот паттерн работает. Иначе выберите другой харнесс.

## Эскиз реализации

Сам драйвер харнесса невелик. Бóльшая часть работы — в двух системных промптах.

```python
# harness.py — pseudocode, omit the error handling for clarity
from pathlib import Path
import json, subprocess, time
from claude_sdk import run_agent

PROJECT = Path("./project")

def initialize(brief: str) -> None:
    """Run once. Turns a brief into a runnable scaffold."""
    run_agent(
        system_prompt=INITIALIZER_PROMPT,
        user_prompt=brief,
        workdir=PROJECT,
        allowed_tools=["write_file", "run_shell", "git"],
        # the initializer is expected to produce:
        #   init.sh, feature_list.json, claude-progress.txt, first git commit
    )
    assert (PROJECT / "feature_list.json").exists()
    assert (PROJECT / "init.sh").exists()

def next_feature() -> dict | None:
    features = json.loads((PROJECT / "feature_list.json").read_text())["features"]
    features.sort(key=lambda f: f["priority"])
    for f in features:
        if f["passes"]:
            continue
        if all(done(dep) for dep in f["depends_on"]):
            return f
    return None  # everything done

def done(feature_id: str) -> bool:
    features = json.loads((PROJECT / "feature_list.json").read_text())["features"]
    return any(f["id"] == feature_id and f["passes"] for f in features)

def code_one_feature() -> bool:
    feat = next_feature()
    if feat is None:
        return False
    run_agent(
        system_prompt=CODING_PROMPT,         # includes the startup ritual
        user_prompt=f"Implement feature {feat['id']}: {feat['title']}",
        workdir=PROJECT,
        allowed_tools=["read_file", "write_file", "run_shell", "git", "puppeteer"],
    )
    # agent is expected to leave a clean commit + updated progress file
    return True

if __name__ == "__main__":
    if not (PROJECT / "feature_list.json").exists():
        initialize(open("brief.md").read())
    while code_one_feature():
        time.sleep(1)  # fresh process each iteration — no state carried over
    print("all features passed.")
```

Две вещи, которые надо усвоить: **каждый вызов `run_agent` — это совершенно новый процесс** — никакой общей памяти, никакого общего окна контекста, память — это файловая система. И **драйвер намеренно туп** — он не выбирает фичи, не валидирует работу и не интерпретирует результаты. Весь интеллект — в промптах и артефактах.

## Завершающая мысль

Этот паттерн работает не потому, что Initializer или кодинг-агент умны. Он работает, потому что **на вопрос «что я знаю?» отвечают файлы на диске, а не содержимое окна контекста.**

Каждый раз, когда вы переносите состояние из внутриконтекстного во внеконтекстное, вы выигрываете в долговечности и не теряете ничего важного. Двухфазный харнесс — это принципиальный способ совершить этот перенос, применённый к конкретной задаче сборки софта на протяжении многих сессий.

Если ваш долгоживущий агент постоянно «забывает», лекарство — почти никогда не более крупное окно контекста. Это больше файлов.
