# Управляемые агенты: разделяем мозг и руки

> **Ключевая идея:** монолитный контейнер агента — где харнесс, песочница и состояние сессии живут в одном процессе — это «питомец», потерю которого вы не можете себе позволить. Разделение мозга (харнесс + LLM), рук (песочница + инструменты) и сессии (журнал событий) превращает каждый компонент в заменяемый «скот» и открывает независимое масштабирование, восстановление после сбоев и изоляцию на уровне безопасности.

## Проблема монолита

Простейшая архитектура агента укладывает всё в одну коробку:

```
┌──────────────────────────────────┐
│         Single Container          │
│                                   │
│  ┌─────────┐  ┌──────────────┐   │
│  │ Harness  │  │   Sandbox    │   │
│  │ (brain)  │  │ (code exec)  │   │
│  │          │  │              │   │
│  │ Session  │  │ Credentials  │   │
│  │ state    │  │ User files   │   │
│  └─────────┘  └──────────────┘   │
│                                   │
└──────────────────────────────────┘
```

Для прототипов это работает. В продакшене такая схема ломается тремя предсказуемыми способами.

### Сбой 1: проблема «питомца»

В аналогии [pets-vs-cattle](https://cloudscaling.com/blog/cloud-computing/the-history-of-pets-vs-cattle/) «питомец» — это именованный, выхоженный вручную сервер, потерю которого вы не можете себе позволить. «Скот» — взаимозаменяем: когда одна особь гибнет, вы просто заменяете её.

Монолитный контейнер агента — это питомец. Если контейнер падает, сессия теряется. Если он перестаёт отвечать, вам приходится выхаживать его обратно к жизни. Вы не можете просто прибить его и поднять новый, потому что состояние сессии — вся история того, что агент уже сделал, — лежит внутри.

### Сбой 2: слепое пятно при отладке

Когда монолитный контейнер зависает, единственное ваше окно наблюдения — внешний поток событий. Но баг в харнессе, потерянный пакет и упавший контейнер выглядят снаружи одинаково. Чтобы диагностировать настоящую причину, инженеру нужен shell-доступ внутрь контейнера — но этот же контейнер хранит пользовательские данные и учётные данные. Отладка превращается в инцидент безопасности.

### Сбой 3: граница безопасности

Недоверенный код, который генерирует агент, выполняется в том же контейнере, что и учётные данные. Внедрению в промпт (prompt injection) достаточно убедить агента прочитать собственные переменные окружения. Как только у атакующего есть эти токены, он может запускать неограниченные сессии. Узкое скоупирование разрешений помогает, но оно кодирует допущения о том, чего модель *не сможет* сделать с ограниченными токенами, — а модели продолжают умнеть.

## Трёхслойная архитектура

Решение — разложить монолит на три независимых слоя, у каждого из которых свой жизненный цикл:

```
┌─────────────────────────────────────────────────────────┐
│                    Orchestration Layer                    │
│                                                          │
│  ┌──────────────┐  ┌──────────┐  ┌───────────────────┐  │
│  │    Brain      │  │ Session  │  │      Hands        │  │
│  │  (Harness +   │  │ (Event   │  │  (Sandbox A)      │  │
│  │   LLM calls)  │  │  Log)    │  │  (Sandbox B)      │  │
│  │              │  │          │  │  (MCP Tools)      │  │
│  │  Stateless   │  │ Durable  │  │  Disposable       │  │
│  └──────┬───────┘  └────┬─────┘  └────────┬──────────┘  │
│         │               │                  │             │
│         │  emitEvent()  │   execute()      │             │
│         ├──────────────►│◄─────────────────┤             │
│         │  getEvents()  │   provision()    │             │
│         ├──────────────►│                  │             │
│         └───────────────┴──────────────────┘             │
└─────────────────────────────────────────────────────────┘
```

### Мозг (харнесс)

Харнесс крутит агентный цикл: вызвать LLM, маршрутизировать вызовы инструментов, управлять контекстом. Принципиально важно: харнесс **не имеет состояния** (stateless). Он не держит данные сессии в памяти. Всё, что должно сохраниться, уходит в журнал сессии через `emitEvent()`.

Когда харнесс падает, поднимается новый, вызывает `wake(sessionId)`, забирает журнал событий через `getEvents()` и продолжает с последнего события. Данные не потеряны. Выхаживать никого не нужно. Харнесс стал «скотом».

### Сессия (журнал событий)

Сессия — это **журнал только на дозапись** (append-only) со всем, что происходило: вызовы LLM, результаты инструментов, пользовательские сообщения, системные события. Она живёт вне и харнесса, и песочницы — в долговременном хранилище.

Ключевые интерфейсы:
- `emitEvent(sessionId, event)` — записать событие в ходе агентного цикла
- `getEvents(sessionId, options)` — прочитать события обратно (позиционная нарезка, фильтрация)
- `getSession(sessionId)` — получить метаданные и статус

Сессия переживает и харнесс, и песочницу. Если любой из них падает, сессия остаётся целой.

### Руки (песочница)

Песочницы — это среды выполнения, где агент запускает код, редактирует файлы и выполняет команды. Они создаются по требованию через `provision({resources})` и уничтожаются, когда больше не нужны.

Харнесс обращается к песочницам так же, как к любому инструменту: `execute(name, input) → string`. Если песочница умирает, харнесс ловит ошибку как неудавшийся вызов инструмента и передаёт её в LLM. Модель может решить повторить (retry) на свежей песочнице.

## Сессия ≠ окно контекста

Это различие — самая тонкая и самая важная часть архитектуры.

**Сессия** — это полная, долговременная запись всего: потенциально миллионы токенов, охватывающие часы или дни работы агента. **Окно контекста** — это подмножество той записи, которое харнесс отбирает для текущего вызова LLM, как правило, 128K–200K токенов.

```
Session (append-only event log, durable)
┌─────────────────────────────────────────────────────────────┐
│ event_1 │ event_2 │ ... │ event_500 │ ... │ event_2000     │
└─────────────────────────────────────────────────────────────┘
                                    │
                          getEvents(slice)
                                    │
                                    ▼
                    Context Window (selected subset)
                    ┌───────────────────────────┐
                    │ system_prompt             │
                    │ event_1950 ... event_2000 │
                    │ (50 most recent events)   │
                    └───────────────────────────┘
```

У этого разделения три преимущества:

**1. Необратимые решения становятся обратимыми.** Уплотнение контекста (compaction — суммаризация старого контекста ради освобождения места) безвозвратно уничтожает информацию. С долговременным журналом сессии харнесс может перечитать любое прошлое событие — даже то, что ранее было выдавлено из окна контекста при уплотнении. Если будущему шагу понадобится информация из event_200, харнесс может её достать.

**2. Инженерия контекста становится заботой харнесса.** Харнесс решает, что попадёт в окно контекста: недавние события, сводку старых событий, системные промпты, релевантное содержимое файлов. Разные реализации харнесса могут использовать разные стратегии, не меняя формат сессии. Сегодняшняя инженерия контекста — это, может быть, обрезка на уровне токенов; завтрашняя — семантический поиск (retrieval). Сессия при этом остаётся прежней.

**3. Становятся возможны многомозговые (multi-brain) архитектуры.** Несколько харнессов могут читать один и тот же журнал сессии одновременно. Мозг-планировщик может просматривать всю историю, пока мозг-исполнитель сосредоточен на недавних событиях. Оба читают из одного и того же долговременного источника.

## Безопасность: учётные данные никогда не попадают в песочницу

Разделённая архитектура естественным образом создаёт границу безопасности. Учётные данные живут в хранилище секретов (vault) вне песочницы. Агент никогда не работает с токенами напрямую.

Два паттерна:

**В связке с ресурсом:** для Git токен доступа используется во время инициализации песочницы — чтобы склонировать репозиторий и настроить локальный remote. `git push` и `git pull` работают изнутри песочницы, и при этом агент ни разу не видит токен.

**Vault + прокси:** для внешних API (через MCP) OAuth-токены лежат в защищённом vault. Агент вызывает MCP-инструменты через прокси. Прокси находит учётные данные сессии в vault и делает внешний вызов. Песочница никогда не видит учётных данных.

```
┌───────────┐     ┌───────────┐     ┌───────────┐     ┌──────────┐
│  Agent    │────►│  MCP      │────►│  Vault    │────►│ External │
│ (sandbox) │     │  Proxy    │     │ (secrets) │     │ Service  │
│           │     │           │     │           │     │          │
│ No creds  │     │ Session   │     │ OAuth     │     │          │
│ here      │     │ → creds   │     │ tokens    │     │          │
└───────────┘     └───────────┘     └───────────┘     └──────────┘
```

Даже если внедрение в промпт убедит агента обыскать всё своё окружение, искать там нечего — учётных данных просто нет.

## Производительность: улучшения TTFT

В монолитном дизайне каждая сессия — даже те, что никогда не касаются песочницы, — оплачивала полную стоимость поднятия контейнера: склонировать репозиторий, запустить процесс, забрать события. Инференс не мог начаться, пока контейнер не готов.

С разделённой архитектурой харнесс стартует сразу. Провижининг песочницы происходит лениво — через вызов инструмента и только при необходимости. Результаты из продакшен-деплоя Anthropic:

| Метрика | Монолит | Разделённая архитектура | Улучшение |
|--------|-----------|-----------|-------------|
| TTFT p50 | базовый уровень | ~40% от базового | **снижение ~60%** |
| TTFT p95 | базовый уровень | ~10% от базового | **снижение ~90%** |

Улучшение на p95 драматическое, потому что наихудший случай старта контейнера (холодные pull-ы, медленный диск) полностью убран с критического пути для сессий, которым песочница не нужна.

## Много мозгов, много рук

Архитектура естественно расширяется в обоих измерениях.

### Масштабирование мозгов

Масштабироваться до множества параллельных сессий агента — значит запускать множество stateless-харнессов. Поскольку каждый харнесс не держит состояния (всё долговременное лежит в журнале сессии), масштабирование горизонтально и тривиально. Никакой разделяемой памяти, никаких распределённых блокировок, никакой синхронизации состояния.

### Масштабирование рук

Один мозг может подключаться к нескольким песочницам. На практике это означает, что агент рассуждает о нескольких средах выполнения и решает, куда отправить работу:

```
┌──────────┐
│  Brain   │
│ (harness)│
└────┬─────┘
     │
     ├──► Sandbox A (Python environment)
     ├──► Sandbox B (Node.js environment)
     ├──► Sandbox C (Customer VPC)
     └──► MCP Tool Server (external APIs)
```

Ранние модели с этим не справлялись — рассуждать одновременно о нескольких средах выполнения было слишком сложно. По мере роста способностей моделей допущение «один контейнер» само стало узким местом. Разделённая архитектура была готова к тому, чтобы модели её догнали.

### Интеграция с VPC заказчика

Когда всё работало в одном контейнере, подключение к инфраструктуре заказчика означало сетевой пиринг — VPC заказчика должен был быть доступен из нашего контейнера. С разделённой архитектурой песочница может работать внутри VPC заказчика, пока мозг работает в нашей инфраструктуре. Мозг общается с песочницей через стандартный интерфейс `execute()` — независимо от того, где песочница физически расположена.

## Антипаттерны

### Хранение состояния сессии в харнессе

```python
# ❌ Session state in harness memory
class HarnessWithState:
    def __init__(self):
        self.events = []  # Lost if harness crashes
        self.current_plan = None
    
    def run_turn(self, message):
        result = self.llm.call(self.events + [message])
        self.events.append(result)  # Only in memory
```

Если харнесс падает, всё пропадает. Всегда эмитьте события в долговременный журнал сессии.

```python
# ✅ Session state externalized
class StatelessHarness:
    def run_turn(self, session_id, message):
        events = self.session_store.get_events(session_id)
        result = self.llm.call(events + [message])
        self.session_store.emit(session_id, result)  # Durable
```

### Учётные данные в песочнице

```python
# ❌ Passing API keys into the sandbox environment
sandbox.execute("export GITHUB_TOKEN=ghp_xxx && git push")
```

Один шаг от утечки через внедрение в промпт. Используйте вместо этого паттерн vault + прокси.

### Жадный провижининг песочницы

```python
# ❌ Provision sandbox before knowing if it's needed
sandbox = provision_sandbox(resources=large_config)  # 30s startup
result = llm.call(prompt)  # Might not even need the sandbox
```

Провижиньте лениво. Пусть LLM сама решит через вызов инструмента, нужна ли ей песочница и какая именно.

## Контрольный список проектирования

Прежде чем строить архитектуру управляемого агента:

1. **Безсостоятельность мозга** — может ли харнесс упасть и перезапуститься без потери данных?
2. **Долговечность сессии** — хранится ли журнал событий вне и мозга, и рук?
3. **Одноразовость песочницы** — можно ли прибить и заменить любую песочницу?
4. **Изоляция учётных данных** — недоступны ли секреты из песочницы?
5. **Ленивый провижининг** — создаются ли песочницы по требованию, а не заранее?
6. **Стабильность интерфейсов** — достаточно ли общи интерфейсы мозг↔руки, чтобы пережить текущие реализации?

## Резюме

Монолитный контейнер агента — естественная отправная точка: всё в одной коробке — это просто. Но он порождает «питомцев», которых нельзя заменить, границы безопасности, которые нельзя обеспечить, и узкие места по производительности, которые нельзя устранить.

Декомпозиция на три слоя — stateless-мозг, долговременная сессия, одноразовые руки — решает все три проблемы. Мозг становится «скотом» (упал — перезапустился). Сессия становится единственным источником истины (переживает всё остальное). Руки становятся ресурсом по требованию (провижинятся только когда нужны, уничтожаются по завершении).

Это тот же паттерн, который операционные системы открыли для себя десятилетия назад: виртуализируй компоненты, сделай интерфейсы стабильными и позволь реализациям свободно меняться внизу. Системному вызову `read()` всё равно, читает ли он с дискового пакета 1970-х или с современного SSD. Интерфейсу `execute()` всё равно, выполняется ли песочница локально, в VPC заказчика или на другом континенте.
