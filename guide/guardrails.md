# Ограничители (Guardrails)

> **Ключевая мысль:** агент без ограничителей — это источник риска. Модель сделает ровно то, что ей сказано, — в том числе то, что ей подсунула prompt-инъекция. Ограничители (guardrails) — это слой разрешений между «модель хочет сделать X» и «харнесс действительно делает X».

## Зачем нужны ограничители

Модель порождает текст. В этот текст входят вызовы инструментов. Харнесс эти вызовы выполняет. А значит, всё, что влияет на вывод модели, способно повлиять и на то, что делает харнесс, — включая вредоносное содержимое в файлах, веб-страницах или сообщениях пользователя.

Это и есть **prompt-инъекция**: злоумышленник встраивает инструкции в данные, которые читает агент, и модель следует этим инструкциям вместо исходной задачи. Без ограничителей prompt-инъекция может:

- Удалить файлы (`rm -rf /`)
- Выкрасть переменные окружения (API-ключи, токены)
- Выполнить произвольный код на хосте
- Отправить несанкционированные сообщения от имени пользователя

Ограничители делают харнесс конечной инстанцией, которая решает, какие действия разрешены, — независимо от того, чего запрашивает модель.

## Модель границы доверия

У любого харнесса есть граница доверия между моделью и операционной средой. Все её пересечения опосредует харнесс:

```
┌──────────────────────────────────┐
│           MODEL SPACE            │
│  (reasoning, tool call requests) │
└────────────┬─────────────────────┘
             │ tool call request
             ▼
┌──────────────────────────────────┐
│         GUARDRAIL LAYER          │
│  Permission check → Allow/Deny   │
└────────────┬─────────────────────┘
             │ approved call
             ▼
┌──────────────────────────────────┐
│        EXECUTION SPACE           │
│  (filesystem, network, shell)    │
└──────────────────────────────────┘
```

Слой ограничителей перехватывает каждый вызов инструмента до его выполнения. Он может:
- **Разрешить (Allow)** — выполнить как запрошено
- **Запретить (Deny)** — вернуть модели ошибку
- **Изменить (Modify)** — переписать вызов (например, ограничить путь к файлу безопасным каталогом)
- **Спросить (Prompt)** — запросить одобрение у человека, прежде чем продолжить

## Модели разрешений

### Белый список (Allow-list, самый строгий)

Разрешены только явно дозволенные действия. Всё остальное запрещено по умолчанию:

```python
ALLOWED_TOOLS = {
    "read_file": {"paths": ["/workspace/**"]},
    "write_file": {"paths": ["/workspace/**"]},
    "run_command": {"commands": ["npm test", "npm run build"]},
}

def check_permission(tool_name: str, args: dict) -> bool:
    if tool_name not in ALLOWED_TOOLS:
        return False
    policy = ALLOWED_TOOLS[tool_name]
    if "paths" in policy:
        return any(fnmatch(args.get("path", ""), p) for p in policy["paths"])
    if "commands" in policy:
        return args.get("command") in policy["commands"]
    return True
```

### Чёрный список (Deny-list, разрешительный)

Разрешено всё, кроме явно заблокированных действий:

```python
BLOCKED_PATTERNS = [
    (r"rm\s+-rf\s+/", "Refusing to delete root filesystem"),
    (r"curl.*\|\s*sh", "Refusing to pipe remote script to shell"),
    (r"env\s+|printenv|echo\s+\$", "Refusing to expose environment variables"),
]

def check_command(command: str) -> tuple[bool, str]:
    for pattern, reason in BLOCKED_PATTERNS:
        if re.search(pattern, command):
            return False, reason
    return True, ""
```

### Многоуровневое одобрение

Разные уровни риска запускают разные процедуры одобрения:

| Уровень риска | Примеры | Действие |
|-----------|----------|--------|
| **Низкий** | Чтение файлов, поиск | Авто-одобрение |
| **Средний** | Запись файлов, прогон тестов | Авто-одобрение с логированием |
| **Высокий** | Выполнение shell-команд, сетевые запросы | Требуется одобрение человека |
| **Критический** | Удаление файлов, push в git, отправка сообщений | Всегда требуется явное одобрение |

```python
def get_risk_level(tool_name: str, args: dict) -> str:
    if tool_name == "read_file":
        return "low"
    if tool_name == "write_file":
        return "medium"
    if tool_name == "run_command":
        cmd = args.get("command", "")
        if any(k in cmd for k in ["rm", "git push", "curl"]):
            return "critical"
        return "high"
    return "medium"
```

## Песочницы (Sandboxing)

Ограничители обеспечивают соблюдение политики; песочницы обеспечивают изоляцию. Песочница (sandbox) — это ограниченная среда исполнения, которая лимитирует то, что код может делать на уровне ОС:

| Технология | Уровень изоляции | Накладные расходы | Сценарий применения |
|-----------|----------------|----------|----------|
| **chroot** | Только файловая система | Минимальные | Базовое ограничение путей |
| **Docker** | Процесс + файловая система + сеть | Низкие | Разработка, CI/CD |
| **Firecracker microVM** | Полноценная VM | Средние | Продакшен с мультиарендностью |
| **gVisor** | Уровень системных вызовов | Низкие-средние | Высокозащищённые нагрузки |
| **WASM** | Уровень языка | Минимальные | Агенты в браузере |

Большинство продакшен-харнессов используют Docker для разработки и Firecracker (или аналог) для продакшена. Ключевой принцип: **исполнение кода агентом ни при каких обстоятельствах не должно иметь доступа к файловой системе, сети или пространству процессов хоста**.

## Очистка входных данных

Помимо ограничителей на уровне инструментов, харнессу следует очищать входные данные, чтобы снизить риск prompt-инъекций:

```python
def sanitize_tool_result(result: str, max_length: int = 50_000) -> str:
    """Truncate and mark external content as untrusted."""
    if len(result) > max_length:
        result = result[:max_length] + "\n[TRUNCATED]"
    # Wrap in markers so the model knows this is external data
    return f"<tool_result>\n{result}\n</tool_result>"
```

Содержимое из внешних источников (веб-страницы, загруженные пользователем файлы, ответы API) должно быть чётко обозначено в контексте, чтобы модель могла отличать инструкции от данных.

## Типичные ошибки

- **Полное отсутствие ограничителей** — норма для большинства любительских харнессов. Годится для локальной разработки, но катастрофично для продакшена.
- **Ограничители только в промпте** — указание модели «не удаляй файлы» ограничителем не является. Модель можно переубедить prompt-инъекцией. Настоящие ограничители реализованы в коде, а не в тексте.
- **Чрезмерно строгие разрешения** — агентом, который не может сделать ничего полезного, никто не будет пользоваться. Нужно балансировать безопасность и пользу.
- **Отсутствие логирования запрещённых действий** — понимание того, что агент *пытался* сделать, но был заблокирован, критически важно для отладки и улучшения промптов.
