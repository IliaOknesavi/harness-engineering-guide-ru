# Что такое харнесс?

> **Ключевая мысль:** модели коммодитизируются — GPT, Claude, Gemini сходятся по возможностям. Настоящий ров (moat) — это харнесс: то, как вы оркеструете контекст, память, инструменты и жизненный цикл агента, определяет, выпустите вы чат-бот или production-агент.

## Определение

**Харнесс** (harness, обвязка / runtime-обёртка) — это runtime-обёртка, которая превращает «голую» языковую модель в **агента** — автономную систему, способную воспринимать своё окружение, принимать решения и совершать действия на протяжении многих шагов ради достижения целей.

Важно отличать «агента» в этом смысле от более раннего употребления термина. В 2023–2024 годах под «агентом» обычно понимали *модель плюс инструменты* — вы давали GPT инструмент веб-поиска и называли это агентом. Агенты, на которые нацелена инженерия харнессов, принципиально сложнее:

| Компонент | «Агент» образца 2023 | Агент эпохи харнессов |
|-----------|-------------|-------------------|
| Модель | ✅ LLM | ✅ LLM |
| Инструменты | ✅ Вызов функций | ✅ Динамическая система инструментов |
| Память | ❌ Без состояния | ✅ Постоянная межсессионная память |
| Управление контекстом | ❌ Наивное | ✅ Сборка контекста по приоритетам |
| Оркестрация | ❌ Один ход | ✅ Агентный цикл с восстановлением после ошибок |
| Среда исполнения | ❌ Хост-процесс | ✅ Изолированный runtime в песочнице |
| Ограничители | ❌ Минимальные | ✅ Модель разрешений + границы доверия |

Харнесс — это инженерный слой, который обеспечивает всё перечисленное. Без него у вас чат-бот, умеющий вызывать функции. С ним — агент, который может ориентироваться в кодовой базе, чинить баги сразу в нескольких файлах и коммитить результат — и всё это автономно.

## Анатомия харнесса

Любой харнесс, независимо от реализации, состоит из четырёх подсистем:

```
┌──────────────────────────────────────────────┐
│                   HARNESS                     │
│                                               │
│  ┌──────────┐  ┌──────────┐  ┌────────────┐  │
│  │ Agentic  │  │   Tool   │  │  Memory &  │  │
│  │   Loop   │  │  System  │  │  Context   │  │
│  └──────────┘  └──────────┘  └────────────┘  │
│                                               │
│  ┌────────────────────────────────────────┐   │
│  │            Guardrails                  │   │
│  └────────────────────────────────────────┘   │
└──────────────────────────────────────────────┘
```

1. **Агентный цикл (Agentic Loop)** — цикл «думать → действовать → наблюдать», который управляет всем поведением агента. Модель рассуждает, вызывает инструмент, наблюдает результат и повторяет цикл, пока задача не будет выполнена.

2. **Система инструментов (Tool System)** — реестр доступных агенту возможностей: файловый ввод-вывод, исполнение команд в shell, веб-поиск, вызовы API. Инструменты могут быть статическими (загружаются при старте) или динамическими (загружаются по требованию через меню навыков).

3. **Память и контекст (Memory & Context)** — система, которая решает, что модель может *видеть*. Сюда входят три самостоятельных аспекта:
   - **Контекст** — то, что попадает в текущий вызов API (системный промпт, инструменты, файлы, история диалога)
   - **Память** — то, что сохраняется между сессиями (MEMORY.md, ежедневные логи, выученные предпочтения)
   - **Сессия** — граница одного запуска агента (история сообщений, результаты инструментов, рабочее состояние-черновик)

4. **Ограничители (Guardrails)** — границы разрешений, обеспечение работы в песочнице и ограничения безопасности. То, что агенту можно и нельзя делать, и то, как не дать инъекции в промпт (prompt injection) обойти эти границы.

Эти четыре подсистемы подробно разбираются в разделе [Базовые концепции](/guide/agentic-loop).

## Минимальный пример

Простейший харнесс — это цикл. Для production он неполон, но структурно верен:

```python
import openai

client = openai.OpenAI()
tools = [{"type": "function", "function": {"name": "read_file", ...}}]

messages = [{"role": "system", "content": "You are a coding agent."}]
messages.append({"role": "user", "content": user_input})

# The agentic loop
while True:
    response = client.chat.completions.create(
        model="gpt-4o", messages=messages, tools=tools
    )
    msg = response.choices[0].message
    messages.append(msg)

    if not msg.tool_calls:
        print(msg.content)  # Done — model has no more actions
        break

    for call in msg.tool_calls:
        result = execute_tool(call.function.name, call.function.arguments)
        messages.append({
            "role": "tool",
            "tool_call_id": call.id,
            "content": result
        })
    # Loop back — model sees the tool results and decides next action
```

Любой харнесс — от 50-строчного скрипта до Claude Code — это вариация этого цикла. Сложность возникает из того, что вы строите *вокруг* него: сборка контекста, сохранение памяти, оркестрация навыков (skill), восстановление после ошибок и изоляция в песочнице.

## Харнесс против фреймворка против runtime

Эти три термина часто путают. Это разные слои:

| Термин | Роль | Примеры |
|------|------|----------|
| **Харнесс** | Оркеструющий код, который оборачивает модель в агента | Claude Code, Codex CLI, OpenClaw |
| **Фреймворк** | Библиотека, дающая строительные блоки для создания харнессов | LangChain, CrewAI, AutoGen |
| **Runtime** | Постоянный процесс, который поддерживает работу харнесса, управляет его жизненным циклом и связывает его с внешним миром | OpenClaw runtime, Docker container, systemd service |

Фреймворк помогает *построить* харнесс. Runtime *хостит* харнесс — поддерживает его «в живых», обрабатывает переподключения, планирует heartbeat-сигналы и маршрутизирует к нему сообщения. Сам харнесс — это логика оркестрации: как собирается контекст, какие инструменты загружаются и как ведёт себя агентный цикл.

## Типичные ошибки

- **Винить модель за проблемы харнесса** — когда агент даёт сбой, обычно дело в контексте (загружены не те файлы, отсутствуют инструкции) или в инструменте (некорректная схема, тихие ошибки), а не в возможностях самой модели.
- **Переусложнять с первого дня** — начните с минимального цикла выше. Добавляйте память, когда понадобится межсессионное состояние. Добавляйте навыки (skill), когда инструментов станет слишком много. Добавляйте ограничители, когда переходите в production.
- **Считать окно контекста безграничным** — модель может рассуждать только о том, что находится в её контексте. Если критически важная информация не собрана в промпт, для модели её фактически не существует.
